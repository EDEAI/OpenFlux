## 1. 产品简介

OpenFlux 是一款开源 AI Agent 桌面客户端，提供多模型接入、长期记忆、浏览器自动化、工具编排等一站式 AI 助手能力。

### 核心特性

| 特性 | 说明 |
|------|------|
| 🧠 多 Agent 路由 | 自动识别用户意图，智能分派到通用/编码/自动化助手 |
| 🔌 多模型支持 | Anthropic / OpenAI / DeepSeek / Moonshot / MiniMax / 智谱 / Google / Ollama |
| 💾 长期记忆 | SQLite + 向量检索（sqlite-vec），支持对话记忆蒸馏与知识沉淀 |
| 🌐 浏览器自动化 | 内置 Playwright，支持网页操作、数据抓取、表单填写 |
| 🛠️ MCP 工具生态 | 兼容 Model Context Protocol，可扩展 Excel、PPT 等外部工具 |
| 🗣️ 语音交互 | 离线语音识别（Sherpa-ONNX）+ Edge TTS 语音合成 |
| 🔒 沙盒隔离 | 本地代码加固 / Docker 容器隔离 |
| 🖥️ 桌面控制 | 键鼠模拟、窗口管理、桌面自动化 |
| 📡 远程访问 | 通过 OpenFlux Router 连接飞书等外部平台 |
| 🌍 国际化 | 中英双语自动切换 |

### 生态定位

```
┌─────────────────────────────────────────────┐
│              NexusAI (企业平台)                │
│  Agent 定义 · 可视化 Workflow · 知识库 · 协作    │
└──────────────────┬──────────────────────────┘
                   │ 标准工作流 / Agent 配置 / API Key 分发
         ┌─────────▼─────────┐
         │  OpenFlux Router  │
         │  集成中枢 · 多端互联 │
         └─────────┬─────────┘
                   │ WebSocket
         ┌─────────▼─────────┐
         │    OpenFlux 桌面   │ ← 你在这里
         │ 本地 Agent · 工具链 │
         └───────────────────┘
```

> **独立使用**：OpenFlux 可以脱离 NexusAI 和 Router 独立运行，只需配置自己的 API Key 即可。

## 2. 系统架构

```
┌─────────────────────────────┐
│       Tauri v2 Shell        │  ← Rust 进程管理 + 原生 API
├─────────────────────────────┤
│     前端 (TypeScript/HTML)   │  ← 聊天 UI / 设置 / 文件预览
├─────────────────────────────┤
│    Gateway Sidecar (Node)   │  ← AI 引擎 / 工具调用 / 记忆系统
└─────────────────────────────┘
```

### 各层职责

| 层 | 技术栈 | 职责 |
|----|--------|------|
| **Tauri Shell** | Rust | 进程生命周期管理、系统托盘、原生窗口、Gateway 进程启停 |
| **前端** | TypeScript + HTML + CSS | 聊天界面、设置面板、文件预览、语音交互 UI |
| **Gateway** | Node.js + TypeScript | AI 引擎核心，包括 LLM 调用、Agent Loop、工具执行、记忆系统、MCP 客户端 |

### 项目目录结构

```
OpenFlux/
├── src/              # 前端 TypeScript（UI / 交互）
│   ├── main.ts       # 前端主入口
│   ├── gateway-client.ts  # Gateway WebSocket 客户端
│   ├── voice.ts      # 语音交互模块
│   ├── i18n/         # 国际化语言包
│   └── styles/       # CSS 样式
├── src-tauri/        # Rust 后端（Tauri Shell）
│   └── src/          # Rust 源码（进程管理、系统托盘、命令层）
├── gateway/          # Gateway Sidecar（AI 引擎）
│   └── src/
│       ├── agent/    # Agent 系统（路由、循环、子Agent、协作）
│       ├── browser/  # 浏览器自动化（Playwright）
│       ├── config/   # 配置解析
│       ├── llm/      # LLM Provider 抽象层
│       ├── tools/    # 内置工具（13 种）
│       ├── workflow/  # 工作流引擎
│       └── utils/    # 工具函数
├── resources/        # 模型文件（嵌入模型等）
├── scripts/          # 构建/重置脚本
├── openflux.yaml     # 主配置文件
└── openflux.example.yaml  # 配置模板
```

## 3. 安装与部署

### 3.1 安装包安装（推荐）

从 [GitHub Releases](https://github.com/EDEAI/OpenFlux/releases) 下载最新安装包（`.msi`），双击安装即可。

安装后首次启动会执行初始化向导，引导配置 LLM API Key。

### 3.2 从源码构建

#### 环境要求

| 依赖 | 最低版本 |
|------|---------|
| Node.js | >= 20 |
| pnpm | >= 10 |
| Rust | stable |
| Tauri CLI | `cargo install tauri-cli --version "^2"` |

#### 构建步骤

```bash
# 1. 克隆仓库
git clone https://github.com/EDEAI/OpenFlux.git
cd OpenFlux

# 2. 安装前端依赖
pnpm install

# 3. 安装 Gateway 依赖
cd gateway && npm install && cd ..

# 4. 配置
cp openflux.example.yaml openflux.yaml
# 编辑 openflux.yaml，填入你的 API Key

# 5. 开发运行
pnpm tauri dev

# 6. 构建安装包
pnpm tauri build
```

#### Gateway 生产构建

使用专用脚本构建 Gateway 生产包：

```powershell
# Windows
.\scripts\build-gateway.ps1

# macOS / Linux
./scripts/build-gateway.sh
```

该脚本执行以下操作：
1. 创建 `gateway-prod/` 目录，复制源码和 `package.json`
2. 使用 npm 安装生产依赖（扁平 `node_modules`，避免 pnpm 深层嵌套）
3. 安装 `tsx`（运行时 TypeScript 执行器）
4. 使用内嵌 `node.exe` 编译 `better-sqlite3` 原生模块，确保 `NODE_MODULE_VERSION` 一致
5. 清理非 win32 平台二进制（onnxruntime / sharp / canvas）
6. 复制嵌入模型到 `resources/`
7. 打包为 `gateway-bundle.tar.gz` 供 Tauri 捆绑

## 4. 配置详解

所有配置集中在 `openflux.yaml` 文件中。首次使用时从 `openflux.example.yaml` 复制。

### 4.1 LLM 供应商配置 (`providers`)

配置各 LLM 供应商的 API Key 和接口地址。

```yaml
providers:
  anthropic:
    apiKey: ${ANTHROPIC_API_KEY}
    baseUrl: https://api.anthropic.com

  openai:
    apiKey: ${OPENAI_API_KEY}
    baseUrl: https://api.openai.com/v1

  minimax:
    apiKey: ${MINIMAX_API_KEY}
    baseUrl: https://api.minimaxi.com/anthropic  # Anthropic 兼容接口

  deepseek:
    apiKey: ${DEEPSEEK_API_KEY}
    baseUrl: https://api.deepseek.com/v1

  zhipu:
    apiKey: ${ZHIPU_API_KEY}
    baseUrl: https://open.bigmodel.cn/api/paas/v4

  moonshot:
    apiKey: ${MOONSHOT_API_KEY}
    baseUrl: https://api.moonshot.cn/v1

  google:
    apiKey: ${GOOGLE_API_KEY}
    baseUrl: https://generativelanguage.googleapis.com/v1beta

  ollama:
    baseUrl: http://localhost:11434/v1  # 本地 Ollama 无需 API Key
```

**支持的供应商及模型概览：**

| 供应商 | 代表模型 | 多模态 |
|--------|---------|--------|
| Anthropic | Claude Opus 4.6, Sonnet 4.5, Haiku 4.5 | ✅ |
| OpenAI | GPT-5, GPT-4.1, o4-mini, o3 | ✅ |
| DeepSeek | DeepSeek Chat (V3.2), R1 | ❌ |
| MiniMax | MiniMax-M2.5, M2.1, M1 | ❌ |
| Google | Gemini 3 Flash, 2.5 Pro/Flash | ✅ |
| Moonshot | Kimi K2.5, K2 Thinking | ✅ (K2.5) |
| 智谱 | GLM-5, GLM-4.6V, GLM-4 Plus | 部分 |
| Ollama | Qwen 2.5, Llama 3.3, LLaVA | 部分 |

### 4.2 LLM 使用配置 (`llm`)

OpenFlux 使用分层 LLM 配置，不同功能可以使用不同模型：

```yaml
llm:
  # 主编排 LLM：用于任务规划和意图理解
  orchestration:
    provider: moonshot
    model: kimi-k2.5
    maxTokens: 4096

  # 执行 LLM：用于工具调用和执行
  execution:
    provider: moonshot
    model: kimi-k2.5
    maxTokens: 4096

  # 嵌入 LLM：用于长期记忆向量化（本地模型，已随安装包打包）
  embedding:
    provider: local
    model: Xenova/paraphrase-multilingual-MiniLM-L12-v2

  # 备用 LLM（可选）：主模型失败时自动切换
  fallback:
    provider: deepseek
    model: deepseek-chat
    temperature: 0.5
```

**嵌入模型选择：**

| 模型 | 大小 | 向量维度 | 语言 | 速度 |
|------|------|---------|------|------|
| `Xenova/paraphrase-multilingual-MiniLM-L12-v2` (默认) | ~120MB | 384 | 中英均衡 | 快 |
| `Xenova/bge-m3` (可选) | ~560MB | 1024 | 多语言最强 | 中 |

> **注意：** 更换嵌入模型后需同步修改 `memory.vectorDim` 配置。

### 4.3 长期记忆配置 (`memory`)

```yaml
memory:
  enabled: true                    # 是否启用长期记忆
  dbName: openflux_memory.db      # 记忆数据库文件名
  vectorDim: 384                   # 向量维度（必须与嵌入模型匹配）
  debug: false                     # 调试模式
  distillation:
    enabled: true                  # 是否启用记忆蒸馏
    startTime: "02:00"             # 蒸馏窗口开始时间
    endTime: "06:00"               # 蒸馏窗口结束时间
    qualityThreshold: 40           # 质量阈值
    sessionDensityThreshold: 5     # 会话密度阈值
    similarityThreshold: 0.85      # 相似度阈值（去重）
```

## 6. 多 Agent 系统

### 6.1 架构概览

OpenFlux 内置多 Agent 路由系统，通过意图分析自动将用户请求分派到最合适的 Agent。

```
用户输入 → Router（意图分析）→ 选择 Agent → Agent Loop（推理+工具调用）→ 返回结果
```

### 6.2 内置 Agent

| Agent ID | 名称 | 职责 | 工具 Profile |
|----------|------|------|-------------|
| `default` | 通用助手 | 一般性问题、闲聊、知识问答、信息查询、翻译 | `full` |
| `coder` | 编码助手 | 编程、文件操作、代码编写、数据处理、文档生成 | `coding` |
| `automation` | 自动化助手 | 浏览器操作、桌面控制、定时任务、工作流自动化 | `automation` |

### 6.3 路由机制

路由分为两条路径：

**快速路径（无 LLM 调用）：**
- 输入过短（< 5 字符）→ 默认 Agent
- `@agentId` 显式指定 → 直接路由
- 仅有一个 Agent → 直接使用
- 关键词匹配（如"买"、"购物"、"浏览器"→ 自动化助手）

**LLM 路径：**
- 将 Agent 列表（id + name + description）发送给 LLM
- LLM 返回最合适的 Agent ID
- 失败时回退到默认 Agent

### 6.4 自定义 Agent

在 `openflux.yaml` 的 `agents.list` 中添加：

```yaml
agents:
  list:
    - id: researcher
      name: "研究助手"
      description: "处理深度研究、资料搜集、论文分析等需要深入检索的任务"
      tools:
        profile: full
        deny:
          - desktop
          - scheduler
```

### 6.5 SubAgent（子 Agent）

Agent 可以派生子 Agent 并发执行子任务：

```yaml
agents:
  defaults:
    subagents:
      maxConcurrent: 5           # 最大并发子 Agent 数
      defaultTimeout: 300        # 默认超时（秒）
      tools:
        deny:                    # 子 Agent 默认禁用的工具
          - scheduler
          - workflow
          - desktop
```

## 7. 工具体系

### 7.1 内置工具列表

OpenFlux 内置 13 种工具，按功能分组：

| 工具组 | 工具名 | 说明 |
|--------|--------|------|
| **文件系统** | `filesystem` | 文件读写、创建、删除、移动、搜索 |
| | `opencode` | 代码执行（Node.js / Python 等） |
| **运行时** | `process` | 进程管理、命令执行 |
| | `spawn` | 子 Agent 派生（并发任务） |
| **Web** | `browser` | 浏览器自动化（Playwright） |
| | `web_search` | Web 搜索（Brave / Perplexity） |
| | `web_fetch` | 网页内容抓取 |
| **系统控制** | `windows` | Windows 系统操作 |
| | `desktop` | 桌面自动化（键鼠模拟、窗口管理） |
| **调度** | `scheduler` | 定时任务管理 |
| | `workflow` | 工作流编排 |
| **办公** | `office` | Office 文档操作 |
| | `email` | 邮件发送 |
| | `notify_user` | 向用户发送通知 |

### 7.2 工具 Profile

预设的工具集合，用于不同场景：

| Profile | 包含工具 | 适用场景 |
|---------|---------|---------|
| `minimal` | 无工具 | 纯聊天 |
| `coding` | filesystem, opencode, process, spawn, office, notify_user | 编码开发 |
| `automation` | browser, web_search, web_fetch, windows, desktop, scheduler, workflow, spawn, notify_user | 自动化任务 |
| `full` | 全部工具 | 通用（默认） |

### 7.3 工具策略（三层过滤）

```
Layer 1: Profile 过滤（按场景裁剪）
    ↓
Layer 2: Agent allow/deny（按 Agent 微调）
    ↓
Layer 3: SubAgent deny（子 Agent 默认禁用危险工具）
```

**配置示例：**

```yaml
# 基于 coding profile，额外允许 web_search，禁用 process
tools:
  profile: coding
  alsoAllow:
    - web_search
  deny:
    - process
```

## 8. MCP 工具扩展

OpenFlux 兼容 [Model Context Protocol (MCP)](https://modelcontextprotocol.io/)，支持接入外部工具服务器。

### 8.1 配置 MCP 服务器

```yaml
mcp:
  servers:
    # Excel 操作工具
    - name: excel
      transport: stdio
      command: uvx
      args: ["excel-mcp-server", "stdio"]

    # PowerPoint 操作工具
    - name: ppt
      transport: stdio
      command: uvx
      args: ["--from", "office-powerpoint-mcp-server", "ppt_mcp_server"]
```

### 8.2 配置参数

| 参数 | 必填 | 说明 |
|------|------|------|
| `name` | ✅ | 服务器名称（唯一标识） |
| `transport` | ✅ | 传输方式：`stdio` 或 `sse` |
| `command` | ✅ | 启动命令 |
| `args` | ❌ | 命令参数列表 |

## 9. 长期记忆系统

### 9.1 工作原理

```
对话消息 → 嵌入向量化 → SQLite + sqlite-vec 存储
                                    ↓
用户提问 → 向量检索相关记忆 → 注入上下文 → LLM 带记忆回答
```

### 9.2 记忆蒸馏

记忆蒸馏是一种后台清理和优化机制：

- **运行窗口**：默认凌晨 02:00 - 06:00（避免影响日常使用）
- **质量阈值**：低于 `qualityThreshold` 的记忆会被清理
- **去重**：相似度超过 `similarityThreshold` 的记忆会被合并
- **密度检查**：会话密度低于阈值的冗余记忆会被清理

### 9.3 嵌入模型

嵌入模型在本地运行（无需联网），默认随安装包打包：

- **默认模型**：`Xenova/paraphrase-multilingual-MiniLM-L12-v2`（384 维，~120MB）
- **高精度模型**：`Xenova/bge-m3`（1024 维，~560MB，需额外下载）

> **重要：** 更换嵌入模型后会触发记忆重建（重新向量化所有已存储的记忆）。

## 12. 沙盒隔离

### 12.1 执行模式

| 模式 | 说明 | 安全级别 |
|------|------|---------|
| `local` | 本地代码加固，限制文件扩展名和目录 | ★★★ |
| `docker` | Docker 容器隔离，完全沙盒 | ★★★★★ |

### 12.2 Local 模式安全措施

- **文件扩展名黑名单**：禁止写入 `.exe`, `.bat`, `.ps1`, `.cmd`, `.vbs`, `.reg`, `.msi` 等
- **目录白名单/黑名单**：通过 `permissions.allowedDirectories` / `blockedDirectories` 控制
- **写入大小限制**：默认最大 50MB (`maxWriteSize: 52428800`)

### 12.3 Docker 模式配置

```yaml
sandbox:
  mode: docker
  docker:
    image: openflux-sandbox       # 镜像名（需先构建）
    memoryLimit: 512m             # 内存限制
    cpuLimit: "1"                 # CPU 限制
    networkMode: none             # none=断网, bridge=允许网络
    cacheVolumes:
      nexus-npm-cache: /root/.npm
      nexus-pip-cache: /root/.cache/pip
    timeout: 60                   # 执行超时（秒）
```

构建沙盒镜像：

```bash
docker build -f Dockerfile.sandbox -t openflux-sandbox .
```

## 17. 常见问题 (FAQ)

### Q1: 最少配置什么可以开始使用？

只需配置一个 LLM 供应商的 API Key 即可。推荐使用 Moonshot（Kimi K2.5，免费额度充足）或 DeepSeek。

### Q2: 嵌入模型需要联网吗？

不需要。默认嵌入模型已随安装包打包在本地，离线运行。

### Q3: 如何更换嵌入模型？

1. 修改 `openflux.yaml` 中的 `llm.embedding.model`
2. 同步修改 `memory.vectorDim`（384 或 1024）
3. 应用会自动检测变更并触发记忆重建

### Q4: Gateway 启动失败？

常见原因：
- Node.js 版本过低（需 >= 20）
- `better-sqlite3` 原生模块未正确编译（`NODE_MODULE_VERSION` 不匹配）
- 端口 18801 被占用

解决方法：
```bash
# 检查 Node.js 版本
node --version

# 重新编译原生模块
cd gateway && npm rebuild better-sqlite3
```

### Q5: 浏览器自动化不工作？

确保：
1. `browser.enabled: true` 已配置
2. Playwright 浏览器已安装：`npx playwright install chromium`
3. 系统有图形界面（无头模式例外）

### Q6: Docker 沙盒如何使用？

1. 安装 Docker Desktop
2. 构建沙盒镜像：`docker build -f Dockerfile.sandbox -t openflux-sandbox .`
3. 修改配置：`sandbox.mode: docker`
4. 重启应用

### Q7: 如何查看日志？

- 运行时日志位于 `logs/` 目录
- 日志格式：`[时间] [级别] [模块] 消息内容`
- Gateway 日志包含 Agent Loop、工具调用、LLM 请求等详细信息

### Q8: 长期记忆数据存在哪里？

- 记忆数据库：`openflux_memory.db`（SQLite）
- 包含 WAL 文件：`openflux_memory.db-wal`、`openflux_memory.db-shm`
- 删除这三个文件即可清空所有记忆

### Q9: 如何接入 Ollama 本地模型？

1. 安装并启动 Ollama: `ollama serve`
2. 拉取模型: `ollama pull qwen2.5:32b`
3. 配置 `openflux.yaml`：
```yaml
providers:
  ollama:
    baseUrl: http://localhost:11434/v1

llm:
  orchestration:
    provider: ollama
    model: "qwen2.5:32b"
  execution:
    provider: ollama
    model: "qwen2.5:32b"
```

### Q10: 数据安全相关

- 所有数据存储在本地，不会上传到任何第三方服务器
- LLM API 调用直接与供应商通信
- 嵌入向量化在本地运行
- 会话、记忆、配置均存储在项目目录或 `~/.openflux/` 下
